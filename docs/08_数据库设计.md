# 数据库设计

## 设计概述

系统采用SQLite数据库，使用SQLAlchemy ORM框架进行数据访问。数据库设计遵循第三范式，确保数据的一致性和完整性。

**设计原则**：
- 数据规范化：避免数据冗余
- 关系完整性：通过外键保证数据一致性
- 性能优化：合理使用索引
- 可扩展性：支持未来功能扩展

## 表结构设计

### 1. users表 (用户表)

**用途**：存储用户基本信息和身份证照片

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name VARCHAR(50) NOT NULL,
    id_number VARCHAR(18) UNIQUE NOT NULL,
    gender VARCHAR(10),
    nation VARCHAR(20),
    birthday DATE,
    address VARCHAR(200),
    id_photo_path VARCHAR(255),
    id_photo_data BLOB,
    face_encoding BLOB,
    face_encoding_timestamp DATETIME,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_users_id_number ON users(id_number);
```

**字段说明**：

| 字段 | 类型 | 约束 | 说明 |
|-----|------|------|------|
| id | INTEGER | PRIMARY KEY | 主键，自增 |
| name | VARCHAR(50) | NOT NULL | 姓名 |
| id_number | VARCHAR(18) | UNIQUE, NOT NULL | 身份证号（唯一） |
| gender | VARCHAR(10) | | 性别（男/女） |
| nation | VARCHAR(20) | | 民族 |
| birthday | DATE | | 出生日期 |
| address | VARCHAR(200) | | 地址 |
| id_photo_path | VARCHAR(255) | | 身份证照片文件路径 |
| id_photo_data | BLOB | | 身份证照片二进制数据 |
| face_encoding | BLOB | | 人脸特征向量（128维） |
| face_encoding_timestamp | DATETIME | | 特征提取时间 |
| created_at | DATETIME | DEFAULT | 创建时间 |
| updated_at | DATETIME | DEFAULT | 更新时间 |

**索引**：
- `idx_users_id_number`：身份证号索引，加速查询

### 2. photos表 (照片表)

**用途**：存储采集的证件照记录

```sql
CREATE TABLE photos (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    photo_type VARCHAR(20) NOT NULL,
    file_path VARCHAR(255) NOT NULL,
    file_size INTEGER,
    photo_spec VARCHAR(20),
    background_color VARCHAR(20),
    quality_score INTEGER,
    face_similarity INTEGER,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_photos_user_id ON photos(user_id);
CREATE INDEX idx_photos_created_at ON photos(created_at);
```

**字段说明**：

| 字段 | 类型 | 约束 | 说明 |
|-----|------|------|------|
| id | INTEGER | PRIMARY KEY | 主键 |
| user_id | INTEGER | FOREIGN KEY | 用户ID（外键） |
| photo_type | VARCHAR(20) | NOT NULL | 照片类型：raw/processed |
| file_path | VARCHAR(255) | NOT NULL | 文件路径 |
| file_size | INTEGER | | 文件大小（字节） |
| photo_spec | VARCHAR(20) | | 证件照规格：一寸/二寸等 |
| background_color | VARCHAR(20) | | 背景色：白色/蓝色等 |
| quality_score | INTEGER | | 质量评分（0-100） |
| face_similarity | INTEGER | | 与身份证照片相似度（0-100） |
| created_at | DATETIME | DEFAULT | 创建时间 |

**索引**：
- `idx_photos_user_id`：用户ID索引
- `idx_photos_created_at`：创建时间索引

**关系**：
- 与users表的一对多关系
- 删除用户时级联删除照片

### 3. collection_records表 (采集记录表)

**用途**：记录采集过程和状态

```sql
CREATE TABLE collection_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    collection_date DATE DEFAULT CURRENT_DATE,
    operator VARCHAR(50),
    status VARCHAR(20) DEFAULT 'pending',
    notes TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_records_user_id ON collection_records(user_id);
CREATE INDEX idx_records_collection_date ON collection_records(collection_date);
```

**字段说明**：

| 字段 | 类型 | 约束 | 说明 |
|-----|------|------|------|
| id | INTEGER | PRIMARY KEY | 主键 |
| user_id | INTEGER | FOREIGN KEY | 用户ID（外键） |
| collection_date | DATE | DEFAULT | 采集日期 |
| operator | VARCHAR(50) | | 操作员名称 |
| status | VARCHAR(20) | DEFAULT | 状态：pending/completed/failed |
| notes | TEXT | | 备注信息 |
| created_at | DATETIME | DEFAULT | 创建时间 |

**索引**：
- `idx_records_user_id`：用户ID索引
- `idx_records_collection_date`：采集日期索引

**关系**：
- 与users表的一对多关系
- 删除用户时级联删除记录

## 数据流程

### 用户采集流程

```
1. 添加用户
   ↓
   INSERT INTO users (name, id_number, ...)
   
2. 采集照片
   ↓
   INSERT INTO photos (user_id, photo_type='raw', ...)
   
3. 处理照片
   ↓
   INSERT INTO photos (user_id, photo_type='processed', ...)
   
4. 记录采集
   ↓
   INSERT INTO collection_records (user_id, status='completed', ...)
   
5. 提取人脸特征
   ↓
   UPDATE users SET face_encoding=..., face_encoding_timestamp=...
```

## 查询示例

### 常用查询

```sql
-- 查询用户及其照片
SELECT u.*, p.* FROM users u
LEFT JOIN photos p ON u.id = p.user_id
WHERE u.id_number = '110101199001011234';

-- 查询采集统计
SELECT 
    COUNT(DISTINCT u.id) as total_users,
    COUNT(p.id) as total_photos,
    COUNT(cr.id) as total_records,
    SUM(CASE WHEN cr.status='completed' THEN 1 ELSE 0 END) as completed,
    AVG(p.quality_score) as avg_quality
FROM users u
LEFT JOIN photos p ON u.id = p.user_id
LEFT JOIN collection_records cr ON u.id = cr.user_id
WHERE cr.collection_date BETWEEN '2026-03-01' AND '2026-03-31';

-- 查询高质量照片
SELECT * FROM photos
WHERE quality_score >= 90
ORDER BY created_at DESC;

-- 查询待处理采集
SELECT * FROM collection_records
WHERE status = 'pending'
ORDER BY collection_date ASC;

-- 查询用户的最新照片
SELECT * FROM photos
WHERE user_id = 1
ORDER BY created_at DESC
LIMIT 1;
```

## 性能优化

### 索引策略

| 表 | 索引 | 用途 |
|-----|------|------|
| users | id_number | 快速查询用户 |
| photos | user_id | 查询用户照片 |
| photos | created_at | 按时间排序 |
| collection_records | user_id | 查询用户记录 |
| collection_records | collection_date | 按日期统计 |

### 查询优化

1. **使用索引**：在频繁查询的字段上建立索引
2. **避免全表扫描**：使用WHERE条件限制结果集
3. **批量操作**：使用事务提高效率
4. **定期维护**：VACUUM清理数据库

### 数据备份

```python
# 定期备份数据库
import shutil
from datetime import datetime

def backup_database():
    timestamp = datetime.now().strftime('%Y%m%d_%H%M%S')
    backup_file = f'id_photo_{timestamp}.db'
    shutil.copy('id_photo.db', backup_file)
```

## 常见问题

### Q1: 如何处理大量数据？

**A**: 使用以下优化方法：
1. 批量插入：使用executemany()
2. 分页查询：使用LIMIT和OFFSET
3. 定期清理：删除过期数据
4. 数据分片：按日期分表

### Q2: 如何保证数据安全？

**A**: 采取以下措施：
1. 定期备份数据库
2. 使用事务保证一致性
3. 设置访问权限
4. 加密敏感数据

### Q3: 如何提高查询性能？

**A**: 优化方案：
1. 添加适当的索引
2. 使用查询优化
3. 避免N+1查询
4. 使用连接池
