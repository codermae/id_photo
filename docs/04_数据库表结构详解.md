# 数据库表结构详解

## 概述

系统采用SQLite数据库，通过SQLAlchemy ORM框架管理4个核心数据表。这4个表通过外键关系形成一个完整的数据模型，支持从采集任务创建、用户管理、照片处理到采集记录追踪的全流程。

## 表结构详解

### 1. 采集任务表 (collections)

**表名**: `collections`

**用途**: 存储采集任务的基本信息，是系统的顶层概念。每个采集任务代表一次完整的证件照采集活动，如"学校A-2026年3月新生入学采集"。

**字段详解**:

| 字段名 | 类型 | 约束 | 设置逻辑 | 说明 |
|--------|------|------|---------|------|
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 系统自动生成 | 采集任务的唯一标识符 |
| `name` | String(100) | NOT NULL | 用户输入 | 采集任务名称，如"学校A-2026年3月"，用于在UI中显示和识别任务 |
| `organization` | String(100) | NOT NULL, INDEX | 用户输入 | 机构名称，如"学校A"、"企业B"，用于分类和统计 |
| `description` | Text | 可选 | 用户输入 | 任务描述，记录任务的详细信息，如采集目的、特殊要求等 |
| `start_date` | Date | 可选 | 默认为当前日期 | 采集开始日期，用于时间范围统计 |
| `end_date` | Date | 可选 | 用户输入 | 采集结束日期，用于时间范围统计 |
| `status` | String(20) | 默认值'active' | 系统管理 | 任务状态，取值为：<br>- `active`: 活跃中（正在进行采集）<br>- `completed`: 已完成（采集工作已结束）<br>- `archived`: 已归档（历史任务） |
| `created_at` | DateTime | 默认为当前时间 | 系统自动生成 | 任务创建时间，用于审计和排序 |
| `updated_at` | DateTime | 默认为当前时间，自动更新 | 系统自动更新 | 任务最后修改时间，每次更新任务时自动更新 |

**关键设计说明**:

1. **name和organization的区别**:
   - `name`: 完整的任务名称，包含机构和时间信息，如"学校A-2026年3月"
   - `organization`: 机构名称，用于分类统计，如"学校A"
   - 这样设计可以支持同一机构的多个采集任务

2. **status字段的状态转换**:
   - 新建任务默认为`active`
   - 采集完成后可改为`completed`
   - 长期保存的历史任务可改为`archived`
   - 状态转换是单向的，不支持回退

3. **时间字段的用途**:
   - `start_date`和`end_date`用于定义采集的时间范围
   - `created_at`和`updated_at`用于审计和追踪任务的生命周期

**关系**:
- 一个采集任务可以包含多个用户（1:n关系）
- 一个采集任务可以产生多条采集记录（1:n关系）

---

### 2. 用户表 (users)

**表名**: `users`

**用途**: 存储需要采集证件照的用户信息。用户是采集任务的参与者，每个用户属于一个采集任务。

**字段详解**:

| 字段名 | 类型 | 约束 | 设置逻辑 | 说明 |
|--------|------|------|---------|------|
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 系统自动生成 | 用户的唯一标识符 |
| `collection_id` | Integer | FOREIGN KEY, NOT NULL, INDEX | 关联采集任务 | 指向采集任务表的外键，确定该用户属于哪个采集任务 |
| `name` | String(50) | NOT NULL | 身份证读取或手动输入 | 用户姓名，从身份证自动读取或手动输入 |
| `id_number` | String(18) | NOT NULL, INDEX | 身份证读取或手动输入 | 身份证号，18位，用于唯一标识用户和验证身份 |
| `gender` | String(10) | 可选 | 身份证读取或手动输入 | 性别，取值为"男"、"女"或"未知"，从身份证号第17位自动提取 |
| `nation` | String(20) | 可选 | 身份证读取或手动输入 | 民族，如"汉"、"回"等，从身份证自动读取 |
| `birthday` | Date | 可选 | 身份证读取或手动输入 | 出生日期，从身份证号自动提取（YYYYMMDD），用于年龄统计 |
| `address` | String(200) | 可选 | 身份证读取或手动输入 | 地址，从身份证自动读取，用于地区统计 |
| `created_at` | DateTime | 默认为当前时间 | 系统自动生成 | 用户创建时间 |
| `updated_at` | DateTime | 默认为当前时间，自动更新 | 系统自动更新 | 用户信息最后修改时间 |

**关键设计说明**:

1. **collection_id的作用**:
   - 确保用户数据的隔离，不同采集任务的用户数据分开存储
   - 支持同一个人在不同采集任务中重复出现（如同一个学生在不同年份的采集）
   - 身份证号在同一采集任务内唯一，但可以在不同任务中重复

2. **身份证号的自动提取**:
   - 性别：从身份证号第17位提取（奇数为男，偶数为女）
   - 出生日期：从身份证号第7-14位提取（YYYYMMDD格式）
   - 民族：从身份证读卡器直接读取
   - 地址：从身份证读卡器直接读取

3. **字段的可选性**:
   - 只有`name`、`id_number`、`collection_id`是必填的
   - 其他字段可选，支持部分信息的导入
   - 系统会自动从身份证号提取性别和出生日期

4. **索引设置**:
   - `collection_id`建立索引，加快按采集任务查询用户的速度
   - `id_number`建立索引，加快身份证号查询的速度

**关系**:
- 多个用户属于一个采集任务（n:1关系）
- 一个用户可以有多张照片（1:n关系）
- 一个用户可以有多条采集记录（1:n关系）

### 3. 照片表 (photos)

**表名**: `photos`

**用途**: 存储采集和处理的照片记录。包括原始照片和处理后的照片，支持多种规格和背景色。

**字段详解**:

| 字段名 | 类型 | 约束 | 设置逻辑 | 说明 |
|--------|------|------|---------|------|
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 系统自动生成 | 照片的唯一标识符 |
| `user_id` | Integer | FOREIGN KEY, NOT NULL, INDEX | 关联用户 | 指向用户表的外键，确定照片属于哪个用户 |
| `photo_type` | String(20) | NOT NULL | 系统设置 | 照片类型，取值为：<br>- `raw`: 原始照片（直接从摄像头拍摄）<br>- `processed`: 处理后的照片（经过裁剪、背景替换、美颜等处理） |
| `file_path` | String(255) | NOT NULL | 系统生成 | 照片文件的完整路径，用于读取和删除照片 |
| `file_size` | Integer | 可选 | 系统计算 | 照片文件大小（字节），用于统计存储空间 |
| `photo_spec` | String(20) | 可选 | 用户选择 | 证件照规格，如"一寸"、"二寸"等，仅对处理后的照片有意义 |
| `background_color` | String(20) | 可选 | 用户选择 | 背景色，如"白色"、"蓝色"、"红色"等，仅对处理后的照片有意义 |
| `quality_score` | Integer | 可选 | 系统计算 | 质量评分（0-100），用于评估照片质量 |
| `face_similarity` | Integer | 可选 | 系统计算 | 与身份证照片的相似度（0-100），用于身份核验 |
| `created_at` | DateTime | 默认为当前时间，INDEX | 系统自动生成 | 照片创建时间，用于排序和统计 |

**关键设计说明**:

1. **photo_type的区分**:
   - `raw`: 原始照片，直接从摄像头拍摄，用于备份和重新处理
   - `processed`: 处理后的照片，经过裁剪、背景替换、美颜等处理，是最终的证件照
   - 一个用户可以有多张原始照片和多张处理后的照片

2. **photo_spec和background_color的用途**:
   - 仅对处理后的照片有意义
   - 支持同一个用户生成多种规格和背景色的照片
   - 用于统计和查询特定规格的照片

3. **质量评分和相似度**:
   - `quality_score`: 评估照片的清晰度、亮度、对比度等
   - `face_similarity`: 评估拍摄照片与身份证照片的相似度，用于身份核验
   - 这两个字段用于质量控制和数据分析

4. **file_path的设计**:
   - 存储完整路径，便于直接读取和删除
   - 支持不同的存储位置（本地、网络等）
   - 可以用于恢复和备份

**关系**:
- 多张照片属于一个用户（n:1关系）

---

### 4. 采集记录表 (collection_records)

**表名**: `collection_records`

**用途**: 记录每个用户的采集过程和状态。用于追踪采集进度、统计采集成功率、生成趋势分析。

**字段详解**:

| 字段名 | 类型 | 约束 | 设置逻辑 | 说明 |
|--------|------|------|---------|------|
| `id` | Integer | PRIMARY KEY, AUTO_INCREMENT | 系统自动生成 | 采集记录的唯一标识符 |
| `collection_id` | Integer | FOREIGN KEY, NOT NULL, INDEX | 关联采集任务 | 指向采集任务表的外键，确定记录属于哪个采集任务 |
| `user_id` | Integer | FOREIGN KEY, NOT NULL, INDEX | 关联用户 | 指向用户表的外键，确定记录属于哪个用户 |
| `collection_date` | Date | 默认为当前日期，INDEX | 系统自动设置 | 采集日期，用于按日期统计和趋势分析 |
| `operator` | String(50) | 可选 | 系统记录或手动输入 | 操作员名称，记录谁进行了采集操作 |
| `status` | String(20) | 默认值'pending' | 系统管理 | 采集状态，取值为：<br>- `pending`: 待采集（已创建记录但未拍照）<br>- `processing`: 待处理（已拍照但照片还在处理中）<br>- `completed`: 已完成（照片已生成） |
| `notes` | Text | 可选 | 系统或手动记录 | 备注，记录采集过程中的特殊情况或问题 |
| `created_at` | DateTime | 默认为当前时间 | 系统自动生成 | 记录创建时间 |

**关键设计说明**:

1. **status字段的状态转换流程**:
   ```
   pending → processing → completed
   ```
   - `pending`: 用户被导入或通过读卡器识别时创建，表示等待拍照
   - `processing`: 用户拍照后，照片进入处理流程时更新为此状态
   - `completed`: 照片处理完成，生成最终证件照时更新为此状态
   - 状态只能向前转换，不能回退

2. **collection_date的用途**:
   - 记录采集的日期，用于按日期统计
   - 支持生成每日采集统计表
   - 支持采集进度趋势分析

3. **采集记录与照片的关系**:
   - 采集记录记录的是采集过程的状态
   - 照片表记录的是具体的照片文件
   - 一条采集记录可以对应多张照片（原始照片和处理后的照片）

4. **多条记录的支持**:
   - 一个用户可以有多条采集记录
   - 支持用户重复采集（如第一次采集失败，第二次重新采集）
   - 每条记录独立追踪采集过程

5. **索引设置**:
   - `collection_id`建立索引，加快按采集任务查询的速度
   - `user_id`建立索引，加快按用户查询的速度
   - `collection_date`建立索引，加快按日期查询的速度

**关系**:
- 多条采集记录属于一个采集任务（n:1关系）
- 多条采集记录属于一个用户（n:1关系）

## 数据流向

### 采集流程中的数据流向

```
1. 创建采集任务
   → 在 collections 表中创建一条记录

2. 导入用户数据
   → 在 users 表中创建多条记录
   → 为每个用户创建一条采集记录（status='pending'）

3. 用户拍照
   → 在 photos 表中创建原始照片记录（photo_type='raw'）
   → 更新采集记录状态为 'processing'

4. 照片处理
   → 在 photos 表中创建处理后的照片记录（photo_type='processed'）
   → 更新采集记录状态为 'completed'

5. 统计报表
   → 查询 collection_records 表统计各状态的用户数
   → 查询 users 表统计人口统计数据
   → 查询 photos 表统计照片数量
```

## 统计查询示例

### 1. 采集进度统计

```sql
-- 查询采集任务的各状态用户数
SELECT 
    status,
    COUNT(*) as count
FROM collection_records
WHERE collection_id = ? AND collection_date BETWEEN ? AND ?
GROUP BY status;

-- 结果示例：
-- completed: 150
-- processing: 30
-- pending: 20
```

### 2. 完成率计算

```
完成率 = 已完成人数 / 用户总数 × 100%

其中：
- 已完成人数 = COUNT(status='completed')
- 用户总数 = COUNT(users) 其中 collection_id = ?
```

### 3. 采集进度三率

```
采集率 = (已完成 + 待处理 + 待采集) / 用户总数 × 100%
处理率 = (已完成 + 待处理) / 用户总数 × 100%
完成率 = 已完成 / 用户总数 × 100%

关系：采集率 ≥ 处理率 ≥ 完成率
```

### 4. 人口统计

```sql
-- 按性别统计
SELECT gender, COUNT(*) as count
FROM users
WHERE collection_id = ?
GROUP BY gender;

-- 按年龄段统计
SELECT 
    CASE 
        WHEN (julianday('now') - julianday(birthday)) / 365 < 18 THEN '0-18'
        WHEN (julianday('now') - julianday(birthday)) / 365 < 25 THEN '18-25'
        -- ... 其他年龄段
    END as age_group,
    COUNT(*) as count
FROM users
WHERE collection_id = ?
GROUP BY age_group;
```

## 数据完整性约束

### 外键约束

1. **users.collection_id** → **collections.id**
   - 删除采集任务时，级联删除所有相关用户

2. **photos.user_id** → **users.id**
   - 删除用户时，级联删除所有相关照片

3. **collection_records.collection_id** → **collections.id**
   - 删除采集任务时，级联删除所有相关采集记录

4. **collection_records.user_id** → **users.id**
   - 删除用户时，级联删除所有相关采集记录

### 业务规则

1. **用户唯一性**:
   - 身份证号在同一采集任务内唯一
   - 支持同一身份证号在不同采集任务中出现

2. **采集记录唯一性**:
   - 一个用户在一个采集任务中可以有多条采集记录
   - 支持用户重复采集

3. **状态转换规则**:
   - 采集记录状态只能向前转换：pending → processing → completed
   - 采集任务状态可以自由转换：active ↔ completed ↔ archived

## 性能优化

### 索引策略

| 表名 | 字段 | 索引类型 | 用途 |
|------|------|---------|------|
| collections | - | - | 表较小，无需额外索引 |
| users | collection_id | 单列索引 | 加快按采集任务查询用户 |
| users | id_number | 单列索引 | 加快身份证号查询 |
| photos | user_id | 单列索引 | 加快按用户查询照片 |
| photos | created_at | 单列索引 | 加快按时间排序 |
| collection_records | collection_id | 单列索引 | 加快按采集任务查询记录 |
| collection_records | user_id | 单列索引 | 加快按用户查询记录 |
| collection_records | collection_date | 单列索引 | 加快按日期查询记录 |

### 查询优化建议

1. **避免全表扫描**:
   - 始终指定 collection_id 进行查询
   - 使用索引字段进行过滤

2. **批量操作**:
   - 使用批量插入而不是逐条插入
   - 使用事务确保数据一致性

3. **缓存策略**:
   - 缓存采集任务列表
   - 缓存统计结果（定期更新）

## 总结

这个数据库设计通过4个核心表实现了完整的采集流程管理：

- **collections**: 采集任务的组织和管理
- **users**: 用户信息的存储和查询
- **photos**: 照片文件的记录和追踪
- **collection_records**: 采集过程的状态管理和统计

通过合理的外键关系、索引设计和状态管理，系统能够高效地支持大规模采集任务，并提供完整的数据分析和统计功能。
